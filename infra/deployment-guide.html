1.2 IAM Bootstrap User / Role

The AWS principal running the CloudFormation deploy needs these managed policies (or equivalent):

  • AWSCloudFormationFullAccess
  • IAMFullAccess — required to create OIDC provider + roles
  • AmazonVPCFullAccess
  • AmazonRDSFullAccess
  • AmazonS3FullAccess
  • AmazonECS_FullAccess
  • AmazonEC2ContainerRegistryFullAccess
  • ElasticLoadBalancingFullAccess
  • CloudFrontFullAccess
  • SecretsManagerReadWrite
Scope down after deploy
This IAM user/role is only needed for the initial CloudFormation deploy. Rotate or delete it afterwards — all ongoing deploys use OIDC roles with minimal permissions.

Deploy CloudFormation Stack

3.1 Stack Deploy Command

Run from the root of the gopng-platform repository. Replace placeholder values with real ones.

aws cloudformation deploy \
  --template-file infra/stack.yml \
  --stack-name gopng-platform \
  --region ap-southeast-2 \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides \
    DBPassword="YourSecurePassword16chars!" \
    PayloadSecretValue="<output of openssl rand -base64 48>" \
    GitHubToken="ghp_xxxxxxxxxxxxxxxxxxxx" \
    GitHubOrg="TMJ-24" \
    GitHubTemplateRepo="site-template" \
    PayloadURL="https://admin.platform.gov.pg" \
    DBInstanceClass="db.t4g.micro" \
    DBMultiAZ="false" \
    CreateOIDCProvider="true"

# Deployment takes ~15 minutes (RDS creation is the long step)
If the OIDC provider already exists
If your AWS account already has a GitHub Actions OIDC provider (from another project), set CreateOIDCProvider=false to skip creation — CloudFormation cannot create a duplicate.

3.2 Retrieve Stack Outputs

aws cloudformation describe-stacks \
  --stack-name gopng-platform \
  --region ap-southeast-2 \
  --query "Stacks[0].Outputs" \
  --output table

First Docker Image & ECS Scale-Up

The ECS service starts with DesiredCount: 0 to avoid a pull failure before any image exists in ECR. The first CI/CD run builds the image, pushes it, and scales the service up to 1. Trigger it manually after configuring secrets:

1
Trigger the deploy workflow On GitHub: Actions → Deploy to AWS ECS → Run workflow → Branch: master.
Or via CLI: 
gh workflow run deploy.yml --repo TMJ-24/gopng-platform --ref master
2
Monitor the workflow The run takes ~8 minutes. Stages: checkout → build Docker image → push to ECR → run DB migration task → update ECS service.
3
Verify the service is running 
aws ecs describe-services \
  --cluster gopng-platform-cluster \
  --services gopng-platform-service \
  --region ap-southeast-2 \
  --query "services[0].{Running:runningCount,Desired:desiredCount,Status:status}"
Expected: {"Running": 1, "Desired": 1, "Status": "ACTIVE"}
4
Test the admin UI Visit the ALBDNSName output in a browser. You should see the Payload CMS login page.
Log in and create the first admin user.

Add S3 Media Storage (Payload Plugin)

Critical — Do Before Going Live
By default, Payload stores uploaded files on the container's local filesystem. On ECS Fargate, the filesystem is ephemeral — files are lost on every task restart or deployment. Install the S3 adapter before any real media is uploaded.

6.1 Install the Package

npm install @payloadcms/storage-s3

6.2 Register the Plugin in payload.config.ts

import { s3Storage } from '@payloadcms/storage-s3'

export default buildConfig({
  plugins: [
    s3Storage({
      collections: {
        media: true,
      },
      bucket: process.env.S3_MEDIA_BUCKET!,
      config: {
        region: process.env.AWS_REGION ?? 'ap-southeast-2',
        // No credentials needed — ECSTaskRole provides them via instance metadata
      },
    }),
  ],
  // ... rest of config
})

Commit and push — the next CI/CD deploy will pick it up. The ECS task role already has s3:PutObject, s3:GetObject, s3:DeleteObject on the media bucket.

DNS & HTTPS Configuration

7.1 Issue ACM Certificate

Request a certificate in ap-southeast-2 (must match the ALB's region):

aws acm request-certificate \
  --domain-name admin.platform.gov.pg \
  --validation-method DNS \
  --region ap-southeast-2

AWS will give you a CNAME record to add to your DNS zone. Add it, wait for status ISSUED:

aws acm list-certificates --region ap-southeast-2 \
  --query "CertificateSummaryList[?DomainName=='admin.platform.gov.pg']"

7.2 Add HTTPS Listener to ALB

# Get the ALB ARN
ALB_ARN=$(aws elbv2 describe-load-balancers \
  --names gopng-alb --region ap-southeast-2 \
  --query "LoadBalancers[0].LoadBalancerArn" --output text)

# Get the target group ARN
TG_ARN=$(aws elbv2 describe-target-groups \
  --names gopng-platform-tg --region ap-southeast-2 \
  --query "TargetGroups[0].TargetGroupArn" --output text)

# Get the cert ARN
CERT_ARN=$(aws acm list-certificates --region ap-southeast-2 \
  --query "CertificateSummaryList[?DomainName=='admin.platform.gov.pg'].CertificateArn" \
  --output text)

# Create HTTPS listener
aws elbv2 create-listener \
  --load-balancer-arn $ALB_ARN \
  --protocol HTTPS --port 443 \
  --certificates CertificateArn=$CERT_ARN \
  --ssl-policy ELBSecurityPolicy-TLS13-1-2-2021-06 \
  --default-actions Type=forward,TargetGroupArn=$TG_ARN \
  --region ap-southeast-2

# Redirect HTTP to HTTPS
aws elbv2 modify-listener \
  --listener-arn $(aws elbv2 describe-listeners \
    --load-balancer-arn $ALB_ARN \
    --query "Listeners[?Port==\`80\`].ListenerArn" --output text \
    --region ap-southeast-2) \
  --default-actions Type=redirect,RedirectConfig='{Protocol=HTTPS,Port=443,StatusCode=HTTP_301}' \
  --region ap-southeast-2

Site Provisioning Flow

8.1 How It Works

When an admin creates a new Site in the Payload CMS admin panel, a fire-and-forget hook (provisionSiteRepo) runs in the background:

1. Admin creates Site
Admin clicks Save in Payload CMS
2. Hook fires async
Admin UI returns immediately — no blocking
3. Repo created
GitHub API clones site-template → site-{slug}
4. Variables set
5 Actions variables written to new repo (slug, API URL, S3 bucket, region, role ARN)
5. Workflow dispatched
GitHub Actions builds Next.js static site and syncs to S3
6. Status updated
Payload Site record updated: deployStatus: deployed, repoName, repoUrl

Disaster Recovery & Rollback

Rollback a Bad ECS Deployment

Every ECS task definition revision is retained in ECR. To roll back to a previous revision:

# List recent task definition revisions
aws ecs list-task-definitions \
  --family-prefix gopng-platform \
  --region ap-southeast-2 \
  --sort DESC

# Roll back to a specific revision
aws ecs update-service \
  --cluster gopng-platform-cluster \
  --service gopng-platform-service \
  --task-definition gopng-platform:REVISION_NUMBER \
  --region ap-southeast-2

Database Snapshot Restore

RDS has deletion protection and automated snapshots (7-day retention). To restore:

# List available snapshots
aws rds describe-db-snapshots \
  --db-instance-identifier gopng-platform-RDSInstance-xxxx \
  --region ap-southeast-2

# Restore to a new instance from snapshot
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier gopng-platform-restored \
  --db-snapshot-identifier rds:snapshot-id \
  --db-instance-class db.t4g.micro \
  --region ap-southeast-2
Update DATABASE_URL after restore
A restored RDS instance has a different endpoint. Update the gopng/database-url Secrets Manager secret and force a new ECS deployment to pick up the new connection string.

CloudFormation Stack Update

To update the infrastructure (e.g. scale up RDS, increase ECS memory):

# Change DBInstanceClass parameter only — no re-deploy needed
aws cloudformation update-stack \
  --stack-name gopng-platform \
  --use-previous-template \
  --parameters \
    ParameterKey=DBInstanceClass,ParameterValue=db.t4g.small \
    ParameterKey=DBPassword,UsePreviousValue=true \
    ParameterKey=PayloadSecretValue,UsePreviousValue=true \
    ParameterKey=GitHubToken,UsePreviousValue=true \
    ParameterKey=GitHubOrg,UsePreviousValue=true \
    ParameterKey=GitHubTemplateRepo,UsePreviousValue=true \
    ParameterKey=PayloadURL,UsePreviousValue=true \
    ParameterKey=DBMultiAZ,UsePreviousValue=true \
    ParameterKey=TaskCpu,UsePreviousValue=true \
    ParameterKey=TaskMemory,UsePreviousValue=true \
    ParameterKey=CreateOIDCProvider,ParameterValue=false \
  --capabilities CAPABILITY_NAMED_IAM \
  --region ap-southeast-2

Post-Deployment Checklist

Work through this list after every fresh deployment. Tick each item before declaring the environment production-ready.

Infrastructure

  • CloudFormation stack status is CREATE_COMPLETE (not ROLLBACK)
  • All stack outputs are populated — copy them to a secure note
  • RDS instance status is available
  • ECS service shows runningCount: 1, desiredCount: 1
  • ALB target group health check is healthy
  • ECR repository has at least one image tagged latest

Application

  • Payload CMS admin loads at ALB DNS name (HTTP first, then HTTPS after cert)
  • First admin user created in Payload
  • /api/health returns 200 OK
  • Media upload: upload a test image in admin — verify it lands in the S3 media bucket
  • Create a test Site record — verify deployStatus changes to building then deployed
  • Visit the CloudFront domain/{site-slug}/ — static site is served

Security

  • RDS is not publicly accessible (verify in AWS Console → RDS → Connectivity)
  • S3 static sites bucket is not publicly accessible directly (only via CloudFront OAC)
  • S3 media bucket objects are accessible publicly (Payload serves them directly)
  • ECS tasks have no public IP (AssignPublicIp: DISABLED)
  • OIDC provider created and both deploy roles are active in IAM
  • GitHub PAT stored in Secrets Manager, not in code or environment
  • Rotate the bootstrap IAM user/role after deploy

DNS & HTTPS

  • ACM certificate issued (status: ISSUED)
  • HTTPS listener added to ALB, HTTP redirects to HTTPS
  • Admin domain CNAME resolves to ALB DNS name
  • Agency domain CNAME resolves to CloudFront domain
  • TLS certificate valid (no browser warning)

Monitoring

  • CloudWatch log group /ecs/gopng-platform receiving logs
  • Set up a CloudWatch alarm on ECS MemoryUtilization > 85%
  • Set up a CloudWatch alarm on RDS FreeStorageSpace < 5GB
  • Set up an ALB 5xx alarm for error rate monitoring
Estimated Monthly Cost (minimum)
  • NAT Gateway: ~$35 USD/mo
  • RDS db.t4g.micro: ~$15 USD/mo
  • ECS Fargate 1 task (1vCPU, 2GB): ~$30 USD/mo
  • ALB: ~$18 USD/mo
  • CloudFront (10GB/mo): ~$1 USD/mo
  • Total: ~$100 USD/mo (light traffic)
Production Hardening (next steps)
  • Enable RDS Multi-AZ (DBMultiAZ=true)
  • Enable ECS Autoscaling (min 1, max 3)
  • Enable WAF on ALB and CloudFront
  • Enable CloudTrail for audit logging
  • Enable RDS Performance Insights
  • Set up AWS Backup plan for RDS
